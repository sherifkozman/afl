I have analyzed the logs and identified several bugs in the project's Firestore database operations. The most critical issue is a crash in the rate-limiting logic due to a missing index specification in the query, while other parts of the system are suffering from performance issues and potential timeouts when dealing with a large number of scan records (over 3 million).

### Findings:
1.  **Crash in `count_scans_since`**: The query for counting recent scans for a client or IP fails because it doesn't specify an ordering that matches the existing Firestore composite indexes, leading to a `google.api_core.exceptions.FailedPrecondition` error.
2.  **Inefficient `count_active_scans`**: This function currently streams ALL scans with an active status. With 3 million total scans, if many are stuck or stale, this results in massive data transfers and "Stream removed" errors (timeouts).
3.  **Bug in `get_latest_active_scan_for_skill`**: This function fetches the 50 oldest scans for a skill and checks if they are active. If a skill has been scanned many times, it will completely miss the currently running (newer) scan.

### Proposed Fix:
I will refactor `backend/db/scans.py` to:
- Add proper `order_by` clauses to match existing composite indexes.
- Add time-based filtering to active scan counts to avoid processing millions of old records.
- Improve query efficiency by pushing more filters to the database layer.
